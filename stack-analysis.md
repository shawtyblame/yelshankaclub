# Анализ и обоснование выбора средств разработки

## Введение

Для реализации веб-системы «Автосервис» — сайта автотюнинг-сервиса — был выбран современный стек технологий. Ниже приведён подробный анализ каждого компонента стека с обоснованием выбора, альтернативами и их сравнением.

---

## 1. Фронтенд

### 1.1 React (v19.2.4)

**Почему выбран:**
- **Компонентный подход** — каждый элемент интерфейса (Header, Footer, Hero, формы) является независимым React-компонентом. Это обеспечивает переиспользование кода, упрощает тестирование и поддержку.
- **Virtual DOM** — React использует виртуальное DOM-дерево, что минимизирует количество операций с реальным DOM и обеспечивает высокую производительность даже при частых обновлениях интерфейса (например, при валидации формы записи в реальном времени).
- **Огромное сообщество и экосистема** — React — самая популярная библиотека для создания UI (по данным State of JS 2024—2025), что гарантирует большое количество готовых решений, библиотек и документации.
- **React 19** — последняя стабильная версия (2025—2026 гг.) с улучшенным серверным рендерингом, поддержкой Actions и оптимизациями производительности.

**Альтернативы и сравнение:**
- *Vue 3* — легче для изучения, но имеет меньшее сообщество и fewer вакансий на рынке. Не подходит для демонстрации навыков работы с индустриальным стандартом.
- *Angular* — избыточно тяжёл для данного проекта (SPA среднего размера); требует TypeScript «из коробки» и более сложной настройки.
- *Svelte* — компилируемый фреймворк, менее распространённый в индустрии.

**Вывод:** React — оптимальный выбор для SPA среднего размера с точки зрения баланса производительности, экосистемы и востребованности на рынке труда.

### 1.2 Vite (v8.0.4)

**Почему выбран:**
- **Мгновенный запуск dev-сервера** — Vite использует нативные ES-модули браузера и не требует сборки всего проекта перед запуском. Сервер стартует за ~200 мс независимо от размера проекта (против 5—30 секунд у Webpack).
- **HMR (Hot Module Replacement)** — изменения в коде отображаются в браузере за <50 мс без перезагрузки страницы, с сохранением состояния компонентов.
- **Оптимизированная сборка** — для production-сборки Vite использует Rollup, который генерирует оптимальный бандл с tree-shaking и code splitting.
- **Современность** — Vite 8 (2026 г.) — самый актуальный сборщик, официально рекомендуемый командой React для новых проектов.

**Альтернативы и сравнение:**
- *Create React App (CRA)* — устарел, официально не рекомендуется с 2024 г., медленная сборка.
- *Webpack* — более гибкий, но требует сложной конфигурации и значительно медленнее в разработке.
- *Next.js* — избыточен для проекта без SSR/SSG.

**Вывод:** Vite — современный стандарт для React-проектов, обеспечивающий максимальную скорость разработки.

### 1.3 React Router DOM (v7.14.0)

**Почему выбран:**
- Стандарт де-факто для маршрутизации в React-приложениях.
- Поддержка вложенных маршрутов, динамических параметров (`/product/:id`), защищённых маршрутов (ProtectedRoute, AdminRoute).
- Версия 7 — последняя стабильная с улучшенной поддержкой загрузчиков данных (loaders).

**Альтернативы:**
- *TanStack Router* — новый, менее распространён.
- *Reach Router* — поглощён React Router.

**Вывод:** React Router — обязательный компонент для SPA с 13 маршрутами и ролевой моделью доступа.

### 1.4 i18next + react-i18next (v26 / v17)

**Почему выбран:**
- Поддержка русского и английского языков без перезагрузки страницы.
- Сохранение выбранного языка в localStorage.
- Версия 26 — актуальная с поддержкой TypeScript, строгих типов и форматов ICU.
- react-i18next — официальный адаптер для React с поддержкой хуков (`useTranslation`).

**Альтернативы:**
- *react-intl* — от FormatJS, более тяжёлый.
- *LinguiJS* — менее популярен.

**Вывод:** i18next — самое популярное решение для интернационализации React-приложений с наибольшим количеством функций.

### 1.5 ESLint 9 (flat config)

**Почему выбран:**
- Статический анализ кода на этапе разработки — предотвращает ошибки до выполнения.
- Flat config (ESLint 9) — новый стандарт конфигурации, более простой и предсказуемый.
- Плагины react-hooks и react-refresh обеспечивают соблюдение правил React и предотвращают ошибки HMR.

**Альтернативы:**
- *Prettier* — только форматирование, не заменяет ESLint.
- *Biome* — новый инструмент, менее зрелый.

**Вывод:** ESLint — обязательный инструмент для поддержания качества кода в командной разработке.

## 2. Бэкенд

### 2.1 Express.js (v5.2.1)

**Почему выбран:**
- **Минималистичность и гибкость** — Express — самый лёгкий и ненавязчивый веб-фреймворк для Node.js. Он не навязывает структуру проекта и позволяет реализовать REST API любой сложности.
- **Огромная экосистема middleware** — cors, multer, morgan, helmet и тысячи других — всё подключается как middleware.
- **Производительность** — Express 5 (2025—2026) основан на async/await из коробки, что обеспечивает эффективную обработку асинхронных запросов.
- **Простота** — 670 строк серверного кода на весь проект (все CRUD-операции, аутентификация, загрузка файлов, email-рассылка) — минимальный порог входа для понимания.
- **Совместимость** — Express 5 поддерживает async error handling без необходимости оборачивать в try-catch каждый роут.

**Альтернативы и сравнение:**
- *Fastify* — быстрее Express (~2x), но менее популярен, меньше готовых middleware. Сложнее найти разработчиков.
- *Koa* — от создателей Express, легче, но требует больше ручной настройки. Версия 2 устаревает.
- *NestJS* — избыточно тяжёл для среднего проекта, требует TypeScript, DI, декораторов. Сложен для дипломного проекта.
- *Hapi* — практически устарел, не обновляется активно.

**Вывод:** Express — оптимальный выбор для REST API среднего размера. Простота, надёжность, огромное сообщество и богатая экосистема middleware делают его лучшим кандидатом.

### 2.2 JSON Web Tokens (jsonwebtoken v9.0.3)

**Почему выбран:**
- **Бессерверная аутентификация** — JWT не требует хранения сессий на сервере (в отличие от session-based auth с Redis/БД). Токен содержит все данные о пользователе (id, email, роль) и проверяется криптографически.
- **Масштабируемость** — JWT позволяет горизонтально масштабировать сервер без потери сессий: любой инстанс может проверить токен, зная секретный ключ.
- **Срок действия** — 7 дней с возможностью обновления.
- **Ролевая модель** — в payload JWT хранится роль (`user` / `admin`), что позволяет реализовать middleware `requireAdmin`.

**Альтернативы:**
- *Session-based (express-session)* — требует Redis или хранения в БД, усложняет масштабирование.
- *OAuth 2.0 / Passport.js* — избыточно для проекта с одной формой входа.

**Вывод:** JWT — стандарт для аутентификации в современных SPA.

### 2.3 bcryptjs (v3.0.3)

**Почему выбран:**
- **Криптостойкое хеширование** — bcrypt использует алгоритм Blowfish с адаптивной сложностью (cost factor = 10). Пароль не хранится в открытом виде даже в случае утечки БД.
- **Встроенная соль** — bcrypt автоматически генерирует уникальную соль для каждого пароля, что делает радужные таблицы бесполезными.
- **Чистый JavaScript** — bcryptjs не требует нативных сборок (в отличие от bcrypt), что упрощает установку на любой платформе (Windows, Linux, macOS).

**Альтернативы:**
- *argon2* — более современный, но требует нативных сборок.
- *SHA-256 + соль* — менее безопасно, подвержено GPU-атакам.
- *scrypt* — реже используется в веб-приложениях.

**Вывод:** bcryptjs — отраслевой стандарт для хеширования паролей в Node.js-приложениях.

### 2.4 Multer (v2.1.1)

**Почему выбран:**
- Стандартный middleware для загрузки файлов в Express.
- Поддержка `multipart/form-data`, ограничения по размеру и количеству файлов.
- Сохранение файлов с уникальными именами (`Date.now() + random`).
- Версия 2.1.1 — актуальная, с поддержкой async/await.

**Альтернативы:**
- *busboy* — низкоуровневый парсер, требует больше ручного кода.
- *formidable* — альтернатива, менее популярна.

**Вывод:** Multer — оптимальный выбор для загрузки изображений в админ-панели.

### 2.5 Nodemailer (v8.0.5)

**Почему выбран:**
- Единственный вменяемый модуль для отправки email в Node.js.
- Поддержка SMTP, HTML-писем, вложений.
- Используется для отправки уведомлений при заполнении формы обратной связи.

**Альтернативы:**
- *SendGrid / Mailgun SDK* — привязаны к конкретным сервисам.
- *Node.js net/smtp* — ручная реализация, нецелесообразна.

**Вывод:** Nodemailer — безальтернативный стандарт для отправки email в Node.js.

### 2.6 CORS (v2.8.6)

**Почему выбран:**
- В разработке фронтенд (Vite:5173) и бэкенд (Express:3001) работают на разных портах — без CORS браузер блокирует запросы.
- Middleware cors() добавляет заголовки `Access-Control-Allow-Origin` и другие.

**Вывод:** Обязательный компонент для клиент-серверной архитектуры с разделёнными портами.

## 3. База данных

### 3.1 PostgreSQL (через pg v8.20.0)

**Почему выбран:**
- **Реляционная модель** — данные сайта (услуги, запчасти, пользователи, заявки) имеют строгую структуру и связи. PostgreSQL — самая мощная открытая реляционная СУБД.
- **Расширяемость** — поддержка JSON (для хранения массива изображений галереи), полнотекстового поиска, транзакций.
- **Параметризованные запросы** — защита от SQL-инъекций на уровне драйвера (`$1`, `$2` вместо конкатенации).
- **Надёжность** — ACID-транзакции используются при создании заявки (одновременное создание записи и списание запчастей со склада).
- **Бесплатность** — PostgreSQL — open-source, не требует лицензий.

**Альтернативы и сравнение:**
- *MySQL / MariaDB* — уступают PostgreSQL в поддержке JSON, транзакций и параллелизма.
- *SQLite (better-sqlite3)* — используется в проекте как вспомогательная БД для тестов/миграций. Не подходит как основная из-за отсутствия конкурентного доступа.
- *MongoDB* — NoSQL, неоптимален для строго структурированных данных с внешними ключами и транзакциями.

**Вывод:** PostgreSQL — лучший выбор для реляционных данных с транзакциями и JSON-полями.

### 3.2 better-sqlite3 (v12.8.0)

**Почему выбран:**
- Используется как вспомогательное хранилище для скриптов миграции и тестов.
- Не требует отдельного сервера БД — работает как встраиваемая БД.
- Синхронный API — упрощает написание скриптов.

**Вывод:** SQLite удобен для локальных миграций и резервного копирования.

## 4. Инструменты разработки и DevOps

### 4.1 npm (Node Package Manager)

Командные скрипты `package.json`:

| Скрипт | Команда | Назначение |
|--------|---------|------------|
| `dev` | `vite` | Запуск frontend dev-сервера на порту 5173 |
| `dev:server` | `node server/index.cjs` | Запуск backend сервера на порту 3001 |
| `dev:all` | `npm run dev:server & npm run dev` | Параллельный запуск обоих серверов |
| `build` | `vite build` | Production-сборка фронтенда в `dist/` |
| `lint` | `eslint .` | Статический анализ всего кода |
| `preview` | `vite preview` | Просмотр production-сборки локально |

**Почему npm, а не yarn/pnpm:**
- npm — стандартный менеджер, входящий в состав Node.js.
- pnpm — быстрее, но менее распространён в учебных заведениях.
- yarn — близнец npm, нет принципиальных преимуществ для данного проекта.

### 4.2 Работа с переменными окружения

В проекте используется гибридный подход:
- Жёстко заданные значения в `db.cjs` и `index.cjs` (для локальной разработки)
- Возможность переопределения через `process.env.PORT`, `process.env.JWT_SECRET`, `process.env.DB_PATH`

## 5. Архитектурные решения

### 5.1 Клиент-серверная архитектура (SPA + REST API)

**Почему выбран такой подход:**
- **Разделение ответственности** — фронтенд отвечает только за UI, бэкенд — за бизнес-логику и данные. Это позволяет независимо разрабатывать, тестировать и развёртывать части системы.
- **Переиспользование API** — REST-эндпоинты могут использоваться не только веб-интерфейсом, но и мобильным приложением или сторонними интеграциями.
- **Гибкость** — фронтенд и бэкенд могут быть написаны на разных технологиях (например, React + Express — классическая связка).

**Схема работы:**
```
Браузер пользователя
    ↓
React SPA (Vite dev:5173)
    ↓ fetch /api/*
Vite proxy (vite.config.js)
    ↓
Express API (port:3001)
    ↓ pg Pool
PostgreSQL (port:5432)
```

### 5.2 JWT-аутентификация

- Пользователь отправляет email + пароль
- Сервер проверяет пароль (bcrypt.compare)
- Сервер возвращает JWT-токен (payload: {id, email, role}, expiresIn: 7d)
- Фронтенд хранит токен в localStorage (через AuthContext)
- При каждом запросе токен отправляется в заголовке `Authorization: Bearer <token>`
- Сервер проверяет токен через middleware authenticateToken

### 5.3 Ролевая модель доступа

- `user` — просмотр каталога, создание заявок, просмотр своих заявок
- `admin` — всё, что может user + управление услугами/запчастями/галереей/заявками/контактами, загрузка файлов

Реализована через middleware `requireAdmin`, проверяющий `req.user.role === 'admin'`.

## 6. Итоговое заключение

Выбранный стек образует **современный, производительный и промышленно применимый набор инструментов**:

| Уровень | Технологии | Обоснование |
|---------|-----------|-------------|
| Фронтенд | React 19 + Vite 8 + React Router 7 + i18next 26 | Современный SPA-стек с максимальной скоростью разработки |
| Бэкенд | Express 5 + JWT + bcrypt + multer + nodemailer | Лёгкий, гибкий, проверенный временем REST API |
| База данных | PostgreSQL + SQLite | Надёжная реляционная СУБД с транзакциями и JSON |
| Качество кода | ESLint 9 (flat config) | Статический анализ, предотвращение ошибок |
| Разработка | npm scripts, Vite proxy, параллельный запуск | Удобный workflow для fullstack-разработки |

**Ключевые преимущества выбранного стека для дипломного проекта:**
1. **Актуальность** — React 19, Vite 8, Express 5 — самые свежие стабильные версии на 2026 год
2. **Промышленная применимость** — точно такой же стек используется в тысячах коммерческих проектов
3. **Полнота** — стек покрывает все потребности: UI, маршрутизация, API, БД, аутентификация, файлы, email, локализация
4. **Масштабируемость** — архитектура позволяет легко добавлять новые модули
5. **Документируемость** — каждый компонент стека имеет исчерпывающую официальную документацию
